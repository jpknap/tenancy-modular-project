## ¿Qué es este proyecto?

Plataforma Laravel 13 multi-tenant usando `stancl/tenancy`. Cada tenant corre en un esquema PostgreSQL aislado. Existen tres proyectos: **Landlord** (admin central), **ActivitiesBoard** (tenant), **SportCompetition** (tenant, mínimo).

El concepto central es un **sistema de CRUD declarativo**: un solo controlador (`AdminController`), un adapter base (`AdminBaseAdapter`) y objetos de configuración (`ListViewConfig`, `CreateViewConfig`, `EditViewConfig`) generan automáticamente todas las vistas de administración. Las rutas se registran desde atributos PHP 8 (`#[Route]`, `#[RoutePrefix]`) — no hay declaraciones manuales de rutas.

---

## Cómo levantar el proyecto

```bash
composer dev        # levanta server + queue + logs + vite en paralelo
composer test       # corre todos los tests
composer analyze    # phpstan + verificación de estilos
```

Accesos:
- Landlord: `http://localhost/landlord/auth/login`
- Tenant (ej. test.localhost): `http://test.localhost/activities-board/auth/login`

**Resumen del flujo:**
1. Admin Landlord hace click en "Acceder al Tenant" → se genera token HMAC (TTL 60s, un solo uso)
2. Redirige a `tenant.localhost/system-access/{token}` → inicia sesión como `system_user`
3. El `system_user` solo ve la pantalla de selección de usuario → hace click en "Suplantar"
4. La sesión cambia al usuario objetivo → aparece banner rojo
5. "Salir de suplantación" → vuelve al `system_user`
6. "Salir del tenant" → invalida la sesión y redirige a `admin.localhost`

## El Sistema de Filtros — Implementación actual

### Arquitectura

Los filtros viven en `app/Common/Admin/Services/Filters/`. Cada estrategia implementa `FilterStrategyInterface`:

```php
interface FilterStrategyInterface
{
    public function applyFilter(Builder $query, string $column, mixed $value): Builder;
    public function render(string $columnName, mixed $currentValue = null): string;
}
```

Estrategias implementadas:
- `TextFilterStrategy` — LIKE sin distinción de mayúsculas ni tildes
- `NumberFilterStrategy` — solo igualdad (`=`), necesita expansión para >, <, entre
- `BooleanFilterStrategy` — `<select>` con Sí/No/Todos
- `DateFilterStrategy` — coincidencia exacta de fecha con `whereDate()`

### Cómo agregar un filtro a una columna (en un Adapter)

```php
use App\Common\Admin\Services\Filters\TextFilterStrategy;
use App\Common\Admin\Services\Filters\BooleanFilterStrategy;
use App\Common\Admin\Services\Filters\NumberFilterStrategy;

// addColumn() retorna $this (el config), no la columna
// Para asignar el filtro hay que obtener la columna primero:
$config->addColumn('name', 'Nombre');
$config->getColumn('name')?->setFilter(TextFilterStrategy::class);
```

### Mínimo de caracteres — estado actual

El mínimo de 3 caracteres está **hardcodeado en el JavaScript** dentro de `resources/views/landlord/list.blade.php` en la línea 189:

```js
if (filterValue.length >= 3) {
    searchParams.set(`filters[${columnName}]`, filterValue);
} else {
    searchParams.delete(`filters[${columnName}]`);
}
```

No existe ningún atributo `data-min-chars` ni propiedad configurable — siempre es `>= 3`. Los filtros Boolean y Date pasan por este mismo JS aunque no necesiten mínimo de caracteres (deberían dispararse inmediatamente en `change`). Es una deuda conocida.

## El Sistema de Ordenamiento — Estado actual

`ListColumn` tiene la propiedad `sortable` y el getter `isSortable()`. La vista de lista renderiza un ícono de orden (`bi-arrow-down-up`) cuando se configura `sortable: true` en una columna, **pero hacer click no hace nada** — no hay lógica de ordenamiento implementada ni en el controlador ni en el JS. Es puramente decorativo.

**Ni `BaseRepository` ni `AdminController` leen un parámetro de orden desde el request.**
